================================================================================
                           FOLDER DESCRIPTIONS
================================================================================

FRONTEND
--------
The frontend folder contains everything for the React single-page application,
built and bundled using Vite.

• public/images/
  All game visual assets - piece sprites for both team colors, dice face 
  images, background artwork, and the Stacks logo.

• src/components/
  Modular, reusable UI building blocks. The board subfolder handles the 
  visual game board, individual tiles, and piece rendering. The dice folder 
  manages dice display and roll animations. The ui folder contains generic 
  interface elements used across multiple pages.

• src/pages/
  Each subfolder represents a distinct route/view in the app. These are 
  container components that compose smaller components into full pages.

• src/hooks/
  Custom hooks that encapsulate reusable logic - managing game state 
  subscriptions, handling authentication flows, wrapping API calls with 
  loading/error states.

• src/context/
  React Context providers for global state that needs to be accessed 
  across many components (current user, active game session, theme).

• src/services/
  The API client layer - functions that communicate with the backend, 
  handle request/response formatting, and manage authentication tokens.

• src/types/
  TypeScript interfaces and type definitions shared across the app - 
  game state shapes, piece types, API payloads, etc.

• src/utils/
  Pure helper functions - coordinate calculations, formatting, 
  validation helpers that don't fit elsewhere.

• src/styles/
  Global CSS, design system tokens (colors, spacing, typography), 
  and any shared style utilities.


BACKEND
-------
The backend folder contains the C# serverless application deployed to AWS Lambda.

• src/Handlers/
  AWS Lambda function entry points. Each handler receives API Gateway events,
  delegates to services, and returns HTTP responses. Kept thin - just 
  request parsing and response formatting.

• src/Core/Models/
  The domain model classes representing game concepts - the board structure,
  piece definitions, player information, move representations.

• src/Core/Services/
  Business logic that enforces game rules - validating that moves are legal,
  checking whose turn it is, detecting victory conditions, rolling dice.

• src/Core/Enums/
  Constants and enumerations - piece types, game phases, player colors, 
  tile states.

• src/Data/Repositories/
  Data access classes that read/write to DynamoDB. Abstracts away the 
  database SDK, handles serialization, provides clean interfaces for 
  storing and retrieving games and users.

• src/Data/Entities/
  Database-specific representations of data - may differ from domain 
  models to optimize for DynamoDB's structure.

• src/Auth/
  Authentication concerns - verifying credentials, generating and 
  validating JWT tokens, password hashing.

• src/Common/
  Cross-cutting utilities - custom exceptions, response helpers, 
  extension methods, logging setup.

• tests/
  Unit tests for game logic, integration tests for data access, 
  and handler tests using mocked dependencies.

• infrastructure/
  AWS SAM templates or CDK stacks that define the cloud resources - 
  Lambda functions, API Gateway routes, DynamoDB tables, IAM roles.


DOCS
----
Project-level documentation - architectural diagrams, API endpoint 
specifications, design system guidelines, and development setup guides.
